# 第一阶段原型文档

## 目标

这一阶段的目标不是做大，而是验证最小可玩闭环是否成立：

- 双武器系统能否提供足够的即时决策
- 敌群波次推进能否形成稳定的战斗节奏
- 局间升级选择能否让单局体验持续产生变化

换句话说，这一版的成功标准是“能从开始一路玩到结束，并且过程中始终有明确的短期目标”。

## 本阶段实现范围

已实现：

- 主菜单
- 开始游戏
- 单个战斗场景
- 玩家移动、受伤、死亡
- 主武器自动朝最近敌人方向射击
- 两种副武器：导弹、震荡弹
- 敌人按波次刷新
- 每波结束后进入局间升级
- 每次升级固定提供 3 选 1
- HUD 展示生命、当前波次、主武器状态、副武器资源
- 死亡结果页，展示生存波次、击毁数、升级历史

明确不做：

- 联机
- 复杂剧情
- 多敌人族群并行设计
- 深层 build 树
- 重美术资源制作
- 大量特效

## 当前战斗闭环

1. 玩家从主菜单进入战斗。
2. 第 1 波敌人开始刷出。
3. 玩家通过移动和主武器维持输出，用副武器处理压力点。
4. 当前波全部清空后，战斗暂停，进入升级 3 选 1。
5. 选择升级后进入下一波。
6. 玩家死亡时结束本局，进入结果页复盘。

这个闭环满足了最小原型最重要的三件事：

- 有即时操作层：移动、主武器、副武器
- 有短周期奖励层：每波升级
- 有整局收束层：死亡结算

## 系统拆分

### 1. 全局状态

`scripts/GameState.gd`

负责记录当前局内的关键数据，包括：

- 当前波次
- 生存波次
- 击毁数
- 已选升级
- 玩家属性
- 副武器资源

这样做的好处是，战斗场景、升级面板和结果页都可以通过同一份状态通信，不会把统计逻辑散落到多个节点里。

### 2. 战斗流程编排

`scripts/BattleController.gd`

这是本阶段的流程中心，负责：

- 开局初始化
- 开始波次
- 按间隔刷敌
- 监听敌人清空
- 进入升级阶段
- 恢复下一波
- 处理玩家死亡并跳结果页

后续如果要加入精英波、事件波或 Boss，也建议继续从这里扩展。

### 3. 玩家与武器

`scripts/Player.gd`

当前玩家设计成尽量简单但可验证：

- 主武器是稳定输出核心
- 副武器是有限资源的强力工具
- `Q` 在导弹和震荡弹之间切换
- `E` 触发当前副武器

这样可以快速观察“切换”和“资源型强技能”是否能提升中程可玩性。

### 4. 波次与升级生成

`scripts/WaveDirector.gd`
`scripts/UpgradeManager.gd`

这两个脚本只负责数据生成，不直接操作场景：

- `WaveDirector` 负责根据波次返回敌人数、生命、速度、接触伤害等参数
- `UpgradeManager` 负责从升级池中随机抽取 3 个候选项

这种写法很适合后续继续把“玩法参数”独立出来。

### 5. UI

`ui/HUD.tscn`
`ui/UpgradeOverlay.tscn`
`scenes/ResultScene.tscn`

UI 被拆成三个阶段：

- 战斗时常驻 HUD
- 波间升级层
- 局终结果页

这样可以保证战斗逻辑和显示逻辑不要混在一起。

## 当前升级池

现有升级都偏“直给”，目的是更快验证节奏，而不是做复杂构筑：

- `Hull Patch`：加生命并回复
- `Rapid Loader`：缩短主武器冷却
- `Thrusters`：提高移动速度
- `Overcharge`：提升主武器伤害
- `Missile Cache`：增加导弹容量并补充
- `Shockwave Supply`：增加震荡弹容量并补充

这批升级的作用是覆盖三种基本收益：

- 生存
- 输出
- 资源

## 已完成验证

项目内提供了自动冒烟验证脚本：

`scripts/SmokeTest.gd`

验证路径为：

- 主菜单加载
- 进入战斗
- 连续完成 3 波
- 每波进入升级选择并完成选择
- 玩家死亡后进入结果页

这可以保证当前仓库至少已经具备“能跑通完整流程”的基础可靠性。

## 下一阶段建议

建议下一阶段仍保持小步推进，优先做这些事情：

1. 给敌人增加 1 到 2 种明确行为差异，而不是直接堆数量。
2. 让副武器和升级之间出现更明显的联动关系。
3. 让不同波次的编排出现轻微节奏变化，例如快波、厚血波、压迫波。
4. 在不增加系统复杂度的前提下补一点击中反馈、受击反馈和波次结束反馈。

如果这几项做完，基本就能开始判断这条 Spacewar II 的核心路线是否值得继续往下扩。
